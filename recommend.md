# 课程设计规划：Web 应用常见安全漏洞分析与防护设计

## 一、选题与核心思路

**选题**：题目5 — Web 应用常见安全漏洞分析与防护设计

**核心思路：通防模式（统一防护中间件）**

大多数 Web 漏洞的根源相同：不可信的用户输入未经验证就进入了敏感操作（SQL 查询、HTML 输出、文件路径、系统命令等）。因此，本设计不采用"逐个漏洞打补丁"的方式，而是在应用层实现一个**统一安全中间件层**，在请求到达业务逻辑之前统一拦截、检测、净化，实现对多类攻击的通用防护。

这本质上是一个轻量级 WAF（Web Application Firewall）的课程级实现，架构上更接近工业实践，论文论述空间也更大。

## 三、业务载体：留言管理系统

漏洞不再是孤立的演示页面，而是嵌入一个具有真实业务场景的**留言管理系统**：

- **普通用户**：注册/登录、发布留言、搜索留言、修改密码、上传头像
- **admin 用户**：以上全部功能 + 用户管理（查看/删除用户）+ 系统工具（ping 检测）

这样每个漏洞都有自然的业务背景，演示更真实，报告论述空间更大。

## 四、覆盖的漏洞类型（10 类）

| # | 漏洞类型 | 所在功能 | 攻击效果 | 通防机制 |
|---|---------|---------|---------|---------|
| 1 | SQL 注入 | 登录、搜索 | 绕过认证、拖库 | 参数化查询 + 关键字检测 |
| 2 | 存储型 XSS | 留言发布 | 持久化脚本执行 | 输出 HTML 编码 + CSP |
| 3 | 反射型 XSS | 搜索结果页 | URL 触发脚本 | 输入净化 + 输出编码 |
| 4 | CSRF | 修改密码、删除留言 | 跨站伪造操作 | CSRF Token 验证 |
| 5 | 路径穿越 | 文件下载 | 读取任意文件 | 路径规范化 + 白名单 |
| 6 | 水平越权 | 留言删除 | 访问/操作他人数据 | 资源归属校验 |
| 7 | 垂直越权 | admin 接口 | 普通用户提权 | 路由角色权限表 |
| 8 | 不安全文件上传 | 头像上传 | 上传可执行文件 | 扩展名白名单校验 |
| 9 | 命令注入 | admin ping 工具 | 执行任意系统命令 | shell 特殊字符过滤 |
| 10 | 暴力破解 | 登录接口 | 枚举密码 | IP 频率限制 + 锁定 |

## 六、通防中间件核心设计（middleware.py）

中间件在 Flask 的 `before_request` 钩子中运行，对每个请求执行以下检查：

```
请求进入
  │
  ├─ 1. 提取所有用户输入（GET参数、POST表单、Cookie、上传文件名）
  │
  ├─ 2. SQL注入检测：匹配关键字模式（OR/UNION/SELECT/DROP等）
  │       → 命中则返回 403，记录日志
  │
  ├─ 3. XSS检测：匹配 <script>、javascript:、on* 事件等
  │       → 命中则净化（HTML实体编码）
  │
  ├─ 4. 路径穿越检测：检查 ../ 序列
  │       → 命中则返回 403
  │
  ├─ 5. 命令注入检测：检查 & | ; ` $( 等 shell 特殊字符
  │       → 命中则返回 403
  │
  ├─ 6. CSRF验证（POST请求）：校验 Token
  │       → 无效则返回 403
  │
  ├─ 7. 越权访问控制：路由权限表 + 资源归属校验
  │       → 越权则返回 403
  │
  ├─ 8. 文件上传校验：扩展名白名单
  │       → 非法扩展名返回 400
  │
  └─ 9. 频率限制：同IP登录失败超阈值
          → 返回 429，锁定 5 分钟
```

响应阶段（`after_request` 钩子）统一注入安全响应头：
- `Content-Security-Policy: default-src 'self'`
- `X-XSS-Protection: 1; mode=block`
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`

## 七、演示流程设计

### 阶段一：漏洞复现（无防护，端口 5000）
1. SQL 注入绕过登录，UNION 注入拖库
2. 存储型 XSS 在留言板持久执行
3. 反射型 XSS 通过 URL 触发
4. CSRF 伪造修改密码
5. 路径穿越读取源码文件
6. 水平越权删除他人留言
7. 垂直越权访问 admin 用户管理
8. 上传 .py 文件绕过类型校验
9. 命令注入执行 whoami
10. 暴力破解登录接口无限尝试

### 阶段二：通防防护（有防护，端口 5001）
1. 启动防护版应用（同样的业务逻辑，加载中间件）
2. 重复上述 10 类攻击，全部被拦截
3. 展示 security.log 拦截记录
4. 展示安全响应头

### 阶段三：对比分析
- 攻击成功率：10/10 → 0/10，防护率 100%
- 中间件代码量 vs 修复每个漏洞的代码量对比
- 讨论通防的局限性（正则绕过风险、误报等）

## 十、关键风险与注意事项

1. **漏洞应用只在本地运行**，不部署到公网，避免被实际利用
2. 通防中间件的**正则检测存在绕过风险**，报告中需诚实讨论局限性（这反而是加分点）
3. 演示视频需包含：姓名学号、代码展示、功能演示、语音解说
4. 提交截止：第17周周二 18:00，压缩包含报告+源代码+可执行程序
5. 查重率需 ≤ 30%，代码注释和报告文字都要原创
