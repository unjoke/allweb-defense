## Context

本课设基于 Python Flask 构建，以**留言管理系统**为业务载体，分普通用户（user）和管理员（admin）两种角色。系统分为两个并行运行的实例：漏洞版（vulnerable，端口 5000）和防护版（protected，端口 5001）。两者共享相同的业务逻辑和模板，区别仅在于防护版加载了通防安全中间件。数据库使用 SQLite，无需额外服务。

**业务功能模块：**
- 登录/注册（普通用户 + admin）
- 留言板（发布、查看、搜索）；留言内容以 `.txt` 文件形式存储在服务器 `messages/` 目录
- 用户管理（admin 专属：查看用户列表、修改用户信息、删除用户）
- 留言文件管理（admin 专属：查看留言文件列表、删除留言文件）
- 文件管理（上传头像/附件、下载文件）
- 个人设置（修改密码、修改个人信息）

## Goals / Non-Goals

**Goals:**
- 实现可复现的 10 类常见 Web 漏洞，覆盖 OWASP Top 10 主要类别
- 实现一个通防中间件，通过 Flask 钩子统一拦截上述攻击
- 提供自动化对比测试，量化防护效果
- 代码结构清晰，便于课设报告引用

**Non-Goals:**
- 不实现完整的生产级 WAF（无机器学习检测、无流量镜像）
- 不覆盖 SSRF、XXE、反序列化等需要复杂环境的高级漏洞
- 不部署到公网

## Decisions

**决策1：使用 Flask `before_request` / `after_request` 钩子实现中间件**
- 理由：Flask 原生支持，无需引入第三方中间件框架，代码量少，逻辑集中，便于报告讲解
- 备选：WSGI 中间件（更底层但复杂度高）、装饰器（需逐路由添加，不符合通防理念）

**决策2：漏洞版和防护版作为两个独立 Flask app，共享 templates/**
- 理由：可以在同一演示中同时启动两个端口，直观对比；共享模板减少重复代码
- 备选：同一 app 用 flag 切换（不够直观，演示效果差）

**决策3：以留言管理系统为业务载体**
- 理由：留言系统天然包含用户输入（留言内容）、权限分级（普通用户/admin）、文件操作、搜索等功能，能自然地承载 10 类漏洞，比单页面演示更贴近真实场景
- 备选：博客系统（功能类似但更复杂）、纯演示页面（缺乏真实感）

**决策4：越权漏洞通过 URL 参数直接传递用户 ID 实现**
- 理由：水平越权（访问他人数据）和垂直越权（普通用户访问 admin 接口）是最常见的访问控制缺陷，通过不校验 session 中的用户身份直接信任参数来实现漏洞
- 防护：中间件层检查 session 角色与请求资源的匹配关系

**决策5：SQL 注入防护使用参数化查询 + 输入关键字检测双重机制**
- 理由：参数化查询是根本修复，关键字检测用于日志记录和演示拦截效果

**决策6：文件上传漏洞通过不校验扩展名和 MIME 类型实现**
- 理由：允许上传 .py/.php 等可执行文件是常见的不安全上传漏洞，防护通过白名单扩展名校验

**决策7：命令注入通过 admin 删除留言文件接口实现**
- 理由：留言以 `.txt` 文件存储在服务器，admin 删除留言文件是自然的业务操作，后端拼接 `subprocess.run("rm messages/" + filename, shell=True)` 是真实场景中常见的错误写法，比独立工具页面更贴近业务
- 攻击方式：攻击者构造 `filename=msg_001.txt; id` 或 `filename=msg_001.txt && cat /etc/passwd` 即可注入任意命令
- 防护：中间件检测 shell 特殊字符（`;`、`&&`、`|`、`` ` ``、`$(`）并拒绝请求

##漏洞与功能模块映射

| 漏洞类型 | 所在功能模块 | 攻击效果 |
|---------|------------|---------|
| SQL 注入 | 登录、留言搜索 | 绕过认证、拖库 |
| 存储型 XSS | 留言发布 | 持久化脚本执行 |
| 反射型 XSS | 留言搜索结果页 | URL 触发脚本 |
| CSRF | 修改密码、删除留言 | 跨站伪造操作 |
| 路径穿越 | 文件下载 | 读取任意文件 |
| 水平越权 | 查看/删除他人留言 | 访问他人数据 |
| 垂直越权 | 普通用户访问 admin 接口 | 提权操作 |
| 不安全文件上传 | 头像上传 | 上传可执行文件 |
| 命令注入 | admin 删除留言文件接口 | 通过文件名注入执行任意 shell 命令（WSL） |
| 敏感信息泄露 | 错误页面、调试接口 | 暴露栈信息/配置 |

## Risks / Trade-offs

- **正则检测可被绕过** → 报告中诚实讨论，作为通防局限性章节内容，不影响课设评分
- **漏洞版应用存在真实漏洞** → 仅绑定 127.0.0.1，不对外暴露
- **CSRF Token 实现较简单** → 使用 Flask session 存储，足够演示用途
- **命令注入在 Windows 上行为与 Linux 不同** → 应用部署在 Linux 环境（或 WSL）下运行，`rm`、`id`、`cat` 等命令均可用；Windows 原生环境下需调整为 `del` 命令，但推荐在 Linux/WSL 下演示
